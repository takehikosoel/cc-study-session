├── testing.md # テスト方針

安全設定をしよう

Claude Codeは実際にファイル削除やコマンド実行ができる。だから制限が必要

Claude Codeに以下を入力
以下の2つを作成してください。

# 1. .claude/settings.json:
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}

# 2. .gitignore に追加:
.env / .env.* / *.pem / *.key
⚠️

ChatGPTと違って、Claude Codeは実際にファイルを削除したりコマンドを実行したりします。便利な反面、誤操作のリスクあり。だから「やっていいこと / ダメなこと」を最初に設定します。

.envファイルって何？

APIキー等の秘密情報を保管する「金庫」

BAD - 絶対ダメ
# コードに直接書く
api_key = "sk-abc123xyz"

→ GitHubに上げた瞬間
  世界中に公開される
GOOD - 正しいやり方
# .envファイル（Gitに上げない）
API_KEY=sk-abc123xyz

# コード本体には：
api_key = os.getenv("API_KEY")
→ 「金庫から取れ」とだけ書く
💡

例え話： .envは金庫。コードには「金庫を開けろ」とだけ書く。鍵（APIキー）をコードに直接書くのは、パスワードを付箋に書いてモニターに貼るのと同じ。

⚠️

Claude Codeに.envを読ませると会話履歴に秘密情報が残ります。だから settings.json で Read(./.env) を禁止して物理的にブロックしています。

11 / 35

各設定の理由

なぜこの設定が必要なのか

何を禁止 なぜ？
.env の読み取り 会話履歴に秘密情報が残る
rm -rf * ファイル一括削除。元に戻せない
git push --force * チームの作業履歴を上書き。全員の作業が消える
.gitignore に .env GitHubに絶対上がらないようにする
✅

設定は1分で終わるが、事故を防ぐ効果は絶大。最初にやるから意味がある。

Skills vs Hooks vs Rules vs Loop

Claude Codeを自動化する4つの仕組み

仕組み いつ動く 何をする 例え
Skills /コマンド名 で呼ぶ or 自動検出 よく使う指示セットを実行 レシピ本
Hooks ツール実行の前後に自動発動 安全チェック・自動フォーマット 火災報知器
Rules セッション開始時に自動読み込み Claude Codeの判断基準を設定 社内ルールブック
Loop 定期間隔で繰り返し実行 定期チェック・監視タスク 定時パトロール
使い分け 判断基準
毎回同じ操作をしている → Skills にまとめる
操作の前後で必ずチェックしたい → Hooks で自動化する
Claude Codeの振る舞いを全体的に変えたい → Rules に書く
定期的に繰り返したい → Loop で自動実行する
💡

Step 3 の deny リストは Hooks の一種。~/.claude/rules/ に置いた development.md や security.md は Rules です。
