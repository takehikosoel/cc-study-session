## なぜ今Claude Codeを学ぶのか

### AIと人間、それぞれの得意なこと

| AIが得意（Claude Codeに任せる） | 人間が得意（あなたがやること） |
|-------------------------------|--------------------------|
| 要件書から設計・コードを生成する | 「何を作るか」を決める（要件・判断） |
| 大量のファイルを一括で作成・編集する | 「これでいいか」を確認する（レビュー・承認） |
| 定型作業の自動化（データ整理、CSV出力等） | 業務知識を伝える（ドメイン知識） |
| 既存コードを読んで構造を把握する | 使い方のルールを決める（セキュリティ・運用方針） |

**今日は「あなたが要件を出して、Claude Codeが設計・実装する」という役割分担を体験します。**

### 世界ではどう使われているか

Claude Codeは「コードを書くツール」ではありません。世界中で**業務全般**に使われ始めています。

**経費精算を30分→5分に（minorun365氏）**
- 「今月の経費精算やって」と話しかけるだけ
- MoneyForward CSVの解析→Gmail領収書検索→科目自動マッピング→コピペ用出力
- 「全自動」にこだわらず「一番面倒な部分だけ自動化」で十分速くなる

**プロダクトマネージャーの日常業務（PM向けコース）**
- PRD（要件定義書）の作成をClaude Codeと壁打ちしながら行う
- 会議メモの整理、データ分析、競合調査をClaude Codeに任せる
- エンジニアでなくてもプロダクト開発に参加できる

**全エンジニアへの導入（カンリー社）**
- 全エンジニアにClaude Codeを導入し、組織として活用レベルを底上げ
- 個人の工夫をチーム資産にする（Skills・ルールの標準化）

**私たちの事例**
- BtoBマーケティングの自動化（メルマガ・追客・ウェビナー企画）
- Instagram月次報告書の自動生成（API→グラフ→スライド→PDF）
- Discord経由でスマホからClaude Codeを操作する仕組み

> 「これをやっていない組織との生産性の差がつきすぎて、変化に適応できない組織はあと数ヶ月もすると一瞬で社会から置いていかれる」 -- minorun365

## 今日作るもの

社内で使う**インフルエンサーデータベース**の初期設計と土台を作ります。

「あなたが要件を出す → Claude Codeが設計・実装する」の流れを体験します：

1. 作業場所を作ってClaude Codeを起動
2. プロジェクトの説明書（CLAUDE.md）を作成
3. 安全設定を入れる
4. 要件書を渡して設計してもらう
5. よく使う操作のショートカットを作る
6. 実装する
7. Gitでコミットする
8. サーバーを起動してブラウザで確認

**完成イメージ:**
```
influencer-db/
├── CLAUDE.md                  # プロジェクトの説明書
├── .claude/
│   ├── settings.json          # 安全設定
│   └── skills/                # ショートカットコマンド
├── models.py                  # データの型定義（インフルエンサー/案件/支払い）
├── app.py                     # Webサーバー
├── seed_data.py               # サンプルデータ
├── requirements.txt           # 依存パッケージ
├── .gitignore                 # 秘密情報をGitから除外
└── README.md                  # 起動手順
```

では始めましょう。

## 1. 作業場所を作って、Claude Codeを立ち上げよう（10分）

### まず「ディレクトリ」を作る

ターミナルを開いてください。これからインフルエンサー図鑑用の**作業フォルダ（ディレクトリ）**を作ります。

ディレクトリとは「フォルダ」のことです。ターミナルではフォルダのことをディレクトリと呼びます。Finderでフォルダを作るのと同じことを、文字で操作します。

```bash
mkdir ~/Desktop/influencer-db
```

これで**デスクトップに `influencer-db` というフォルダが作られました**。Finderで確認してみてください。

次に、そのフォルダに移動して、Gitを初期化します：

```bash
cd ~/Desktop/influencer-db
git init
```

| コマンド | 何をした？ | Finderで言うと |
|---------|----------|-------------|
| `mkdir ~/Desktop/influencer-db` | デスクトップにフォルダを作った | 右クリック→新規フォルダ |
| `cd ~/Desktop/influencer-db` | そのフォルダに移動した | フォルダをダブルクリックして開いた |
| `git init` | 変更履歴の記録を開始した | （Finderにはない機能。「いつ誰が何を変えたか」を記録する仕組み） |

### Claude Codeを起動する

```bash
claude
```

`>` が表示されたら起動成功です。

### そもそもClaude Codeって？

みなさんが使っているChatGPTやClaude.aiは「ブラウザで質問→テキストが返る」ツールです。回答をコピペして自分で作業しますよね。

**Claude Codeは違います。** 今作ったフォルダの中で、実際にファイルを作ったり、コードを書いたり、コマンドを実行してくれるAIです。

| | ChatGPT / Claude.ai | Claude Code |
|--|---------------------|-------------|
| どこで使う | ブラウザ | ターミナル（今開いている画面） |
| AIの役割 | 質問に答える | 実際にファイルを作って操作する |
| 例えるなら | 電話相談 | 隣に座ってPC操作してくれる同僚 |

### まず話しかけてみよう

「このフォルダの中身を教えて」と入力してみてください。

まだ空っぽですが、これからClaude Codeと一緒にファイルを作っていきます。

### 覚えておくコマンド5つ

作業中に使うコマンドです。今は覚えなくて大丈夫、必要なタイミングで都度案内します。

| コマンド | いつ使う |
|---------|---------|
| `Esc` | Claude Codeを止めたいとき |
| `Esc` x2 | やり直したいとき（巻き戻し） |
| `/clear` | 新しい作業に入るとき（会話リセット） |
| `/context` | 記憶容量の使用状況を確認 |
| `/plan` | 調査だけしたいとき（読み取り専用モード） |

### Claude Codeに「何を作るか」を伝える

Claude Codeに以下を入力してください：

```
以下の内容でCLAUDE.mdを作成してください。

# Influencer Database

## Goal
社内で使うインフルエンサーデータベース。
インフルエンサー情報を会社として一元管理し、
担当者個人に依存しがちなキャスティング・進行・支払い管理をチームで見える化する。
「インフルエンサー個人ベース」で情報が蓄積される構造にする。

## Tech Stack
- Python 3.11+ / SQLite（ローカルDB）
- 日本語UIのWebアプリ
- CSVインポート・エクスポート対応

## 主要機能
- インフルエンサー管理（一覧/検索/詳細/登録/編集）
- 案件管理（インフルエンサーに紐づく）
- 支払い管理（承認フロー付き）
- ナレッジ管理（社内知見の蓄積）
- 担当者管理・権限管理

## Directory Structure
（Claude Codeに提案させてください）

## Rules
- 機密情報は .env で管理（コードにハードコードしない）
- 個人情報を扱うため表示制御を考慮する
- テスト: pytest で実行
- 日本語で分かりやすいUI
```

**許可画面が出たら `y` を押してください。** これがClaude Codeの安全設計です。ファイルを作る前に必ず確認してきます。

> **Tips: 許可画面の隠し技**
> y/n だけでなく、**Tabキーを押すと追加指示を入力できます**。
> 例：`y` → `Tab` → 「ディレクトリ構成をもう少し詳しく」

### ちなみに: CLAUDE.mdって何？

今作った `CLAUDE.md` は、Claude Codeへの「プロジェクトの説明書」です。

**Claude Codeは毎回起動するときにこのファイルを最初に読みます。** だから毎回「このプロジェクトは〇〇で...」と説明しなくて済みます。新しい人への引き継ぎメモのようなものです。

**書き方のコツ：**
- **50行以内**に収める（長すぎるとClaude Codeが重要なルールを見落とす）
- **Goal / Tech Stack / Directory / Rules** の4セクションが基本
- コードを見ればわかることは書かない

> 実際に運用中のプロジェクトの説明書を見てみましょう：
> ```bash
> cat /Users/satoutakehiko/Desktop/cc_folder/soel-btob-marketing/CLAUDE.md
> ```
> 35行で収まっています。これがお手本です。

## 3. 安全設定をしよう（15分）

### なぜ安全設定が必要？

Claude Codeは ChatGPT と違って、**実際にファイルを削除したりコマンドを実行したりします**。便利な反面、誤操作のリスクもあります。だから「やっていいこと / ダメなこと」を最初に設定します。

### Claude Codeに安全設定を作ってもらう

以下を入力してください：

```
以下の2つを作成してください。

1. .claude/settings.json:
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}

2. .gitignore に以下を追加:
.env
.env.*
*.pem
*.key
```

### ちなみに: .envファイルって何？

インフルエンサーDBを作るには、InstagramのAPIキーやGoogleの認証情報が必要になります。これらの**秘密情報**を保管するのが `.env` ファイルです。

```
コードに直接書く（絶対ダメ）：
  api_key = "sk-abc123xyz"
  → GitHubに上げた瞬間、世界中に公開される

.envファイルに書く（正しいやり方）：
  API_KEY=sk-abc123xyz       ← このファイルはGitに上げない
  コード本体には：
  api_key = os.getenv("API_KEY")  ← 「金庫から取れ」とだけ書く
```

**例え話：** .envは金庫。コードには「金庫を開けろ」とだけ書く。鍵（APIキー）をコードに直接書くのは、パスワードを付箋に書いてモニターに貼るのと同じです。

### なぜClaude Codeに.envを読ませない？

Claude Codeに読ませると会話履歴に秘密情報が残ります。だから `settings.json` で `Read(./.env)` を禁止して物理的にブロックしています。

### 各設定の理由

| 何を禁止 | なぜ？ |
|---------|-------|
| `.env` の読み取り | 会話履歴に秘密情報が残る |
| `rm -rf *` | ファイル一括削除。**元に戻せない** |
| `git push --force *` | チームの作業履歴を上書き。**全員の作業が消える** |
| `.gitignore` に `.env` | GitHubに絶対上がらないようにする |

### ちなみに: 3層の安全設計

Claude Codeの安全は3層で守られています：

```
第1層: settings.json の禁止リスト → 物理的にブロック
第2層: ルールファイル（rules/）→ そもそも危険な操作を提案しなくなる
第3層: 許可ダイアログ（y/n）→ 最後の砦。あなたが判断する
```

「1つが抜けても他で止まる」設計です。

> 実際に使っている安全設定を見てみましょう：
> ```bash
> cat ~/.claude/settings.json
> cat ~/.claude/rules/security.md
> ```

## 4. どんなデータベースにするか設計しよう（15分）

### いきなり作らない。まず設計する

ここでClaude Codeの「Plan Mode」を使います。**読み取り専用モード**で、ファイルを変更せずに調査や設計だけしてもらえます。

**`/plan` と入力して Plan Mode に入ってください。**

Plan Modeに入ったら、以下の要件を**そのまま丸ごと**Claude Codeに貼り付けてください。長いですが大丈夫です：

```
以下の要件でインフルエンサーデータベースを設計してください。
いきなり実装せず、まず設計を提案してください。

■目的
- インフルエンサー情報を会社として一元管理したい
- 担当者個人に依存しがちなキャスティング・進行・支払い管理をチームで見える化したい
- 過去案件の実績やナレッジを蓄積したい
- 支払い漏れや対応漏れを防ぎたい

■設計思想
- 「インフルエンサー個人ベース」で情報が蓄積される構造
- インフルエンサーに対して、過去案件実績・進行中案件・支払い情報・担当者・社内ナレッジが紐づく

■必要な機能
1. インフルエンサー管理（一覧/検索/詳細/登録/編集）
   管理項目: 名前/活動名, SNSアカウント(Instagram/TikTok), フォロワー数,
   エンゲージメント率, フォロワー属性, ジャンル, 所属事務所, 連絡先, 社内メモ

2. 案件管理（インフルエンサーに紐づけ）
   管理項目: クライアント名, 案件名, ステータス(依頼中/進行中/投稿待ち/投稿完了),
   担当者, 投稿日, 金額, 成果メモ

3. 支払い管理（承認フロー付き）
   ステータス: 下書き→申請中→上長確認待ち→承認済み→支払い完了 or 差し戻し
   支払い条件: 投稿完了+URL確認+請求書受領+条件達成+上長承認 → すべて揃って支払い可能

4. ナレッジ管理（インフルエンサーごとの社内知見）
   相性の良い案件ジャンル, 注意点, レスポンス速度, 投稿クオリティ, 再依頼可否

5. 権限管理
   一般担当者: 登録・編集・申請 / 上長: 承認・差し戻し / 管理者: 全権限

■技術要件
- サーバー上で動くWebアプリ / 日本語UI / SQLite / CSVインポート・エクスポート

■提案してほしいこと
1. 技術構成
2. データベース設計（テーブル構成）
3. 画面構成
4. 開発ステップ

コードは書かないでください。設計だけ出してください。
```

**ポイント：** 要件書をそのままClaude Codeに渡しています。非エンジニアでも「やりたいこと」を日本語で書ければ、Claude Codeが技術設計に翻訳してくれます。

### ちなみに: なぜ「まず設計」なのか

> 「直接コーディングにジャンプさせると、間違った問題を解決するコードが生成される」 -- 公式ベストプラクティス

いきなり「作って」と頼むと、Claude Codeは間違った方向に全力で走ることがあります。

自社では **PSB** という3段階で進めています：

| フェーズ | 今日の作業で言うと | なぜ？ |
|---------|------------------|-------|
| **P**lan（設計） | 今やっているところ（Step 4） | いきなり作ると間違った方向に走る |
| **S**etup（準備） | CLAUDE.md + 安全設定（Step 2-3） | 最初の手間が後で何倍にもなって返ってくる |
| **B**uild（実装） | この後やる（Step 6-7） | 設計通りに作る |

### Plan Modeの承認画面

設計が出たら、承認画面が表示されます。**「Yes, clear context and auto-accept edits」**を選ぶのがおすすめです。

| 選択肢 | 意味 |
|-------|------|
| **Yes, clear context...（推奨）** | 調査で溜まった情報をクリアして、計画だけ残して実行 |
| Yes, auto-accept edits | そのまま実行 |
| Yes, manually approve | 一つずつ確認しながら |

設計段階でClaude Codeが大量のファイルを読み込んでいますが、実装時にはこれがノイズになります。「clear context」で**計画だけ残してクリーンに**します。

### 3つのモードまとめ

| モード | 操作 | 何ができる | いつ使う |
|-------|------|----------|---------|
| 通常モード | そのまま起動 | 読み書き両方 | 普段使い |
| Plan Mode | `/plan` | 読み取りだけ | 設計・調査 |
| Auto Mode | 設定で切替 | 自動で進む | 信頼できるタスクを一気に |

## 5. 記憶をリセットして実装に備えよう（5分）

### /clear してリフレッシュ

設計を確認したら、**`/clear` を実行**してください。

### なぜリセットするの？

Claude Codeには「記憶容量」があります。会話が長くなると、**初期の指示を忘れたり、間違いが増えたりします**。

95%を超えると自動圧縮されて、「〇〇しないで」という禁止事項が消えてしまうこともあります。

**だから作業の区切りごとに `/clear` する。** これが一番大事なテクニックです。

`/context` を実行すると、今どれくらい記憶容量を使っているか確認できます。

| テクニック | いつ使う |
|-----------|---------|
| `/clear` | 新しい作業に入るとき |
| `/context` | 記憶容量の使用状況を確認 |
| Document & Clear | 長い作業の途中で進捗をファイルに書き出してから /clear |

**今日のルール：各ステップの間で「記録 → /clear」します。**

`/clear` する前に、必ずClaude Codeに進捗を記録させてください：

```
ここまでの進捗と決定事項を progress.md に書き出してください。
次のセッションで読み返せるように、何を作ったか・何が決まったかをまとめてください。
```

書き出しが終わったら `/clear` → 次のステップの最初に `@progress.md を読んでから作業してください` と伝える。

これが「**Document & Clear**」です。記憶をリセットしても、ファイルに書いてあるから引き継げます。

   - payments テーブル（支払い管理。承認ステータス付き）

### ちなみに: このプロジェクトで使う「API」とは

実装の中で「API」という言葉が出てきます。ここで整理しておきます。

**APIとは？**

API = 「あるサービスの機能を、外部のプログラムから利用するための窓口」です。

例え話：レストランの「注文カウンター」がAPIです。お客さん（プログラム）がカウンター（API）に「カレーください」と伝えると、厨房（サービス）が料理を作って返してくれます。直接厨房に入る必要はありません。

**このプロジェクトで使うAPI：**

| API | 誰が提供 | 何ができる | なぜ必要 |
|-----|---------|----------|---------|
| **Instagram Graph API** | Meta社 | インフルエンサーのフォロワー数、投稿データ、エンゲージメント率を取得 | インフルエンサー情報を自動収集するため |
| **Claude API** | Anthropic社 | AIにテキスト生成・分析をさせる | コメント分析や候補推薦の自動化 |
| **Google Sheets API** | Google | スプレッドシートの読み書き | データの保存・共有 |

**APIキーとは？**

APIを使うには「鍵」が必要です。これが**APIキー**です。レストランの会員カードのようなもので、「この人は利用許可がある」と証明します。

だから Step 3 で `.env` に保管し、コードに直接書かないようにしたのです。

**LLM APIとは？**

Claude Code自体も裏で**LLM API**（大規模言語モデルのAPI）を使っています。あなたがClaude Codeに話しかけるたびに、裏でAnthropic社のAPIにリクエストが送られ、AIの回答が返ってきます。

```
あなた → Claude Code → [LLM API] → Anthropic社のAIモデル → 回答
```

これがStep 5で学んだ「記憶容量」と関係します。会話が長くなるとAPIに送るデータが増え、精度が下がるのです。

### ちなみに: API vs MCP（もう一つの連携方法）

Claude Codeには、APIとは別に**MCP**（Model Context Protocol）という連携方法もあります。

| | API | MCP |
|--|-----|-----|
| 何が起きる | コード（プログラム）が生成される | Claude Codeが外部サービスに直接アクセスする |
| Claude Code閉じたら？ | **コードが残るから独立して動き続ける** | **Claude Codeが起動していないと使えない** |
| 例えるなら | レシピを書いてもらう→次から自分で作れる | 専属シェフを雇う→いる間は何でも頼めるが、帰ったら作れない |
| APIキーの管理 | `.env`に保管してコードから参照 | Claude Codeの設定に登録 |

**APIはClaude Codeなしでも独立して動くプログラムになります。** MCPはClaude Codeに外部サービスへの「接続口」を追加する仕組みで、設定すれば毎回使えますが、Claude Codeが起動していないと機能しません。

| やりたいこと | どっち？ | 理由 |
|------------|---------|------|
| インフルエンサー情報を毎日自動収集 | **API** | 一度コードを作れば毎日自動で動く |
| 今すぐDBの中身を確認したい | **MCP** | 「見せて」で即回答 |
| 毎週メルマガ自動送信 | **API** | Claude Codeなしで動き続ける |
| Discordで通知を受け取りたい | **MCP** | Claude Code起動中だけ |

今回のプロジェクトでは**API方式**を使います。コードとして残るので、一度作ればClaude Codeなしでも動き続けます。

2. **「.envは作成しない」** → 安全ルールを明示（Step 3 で設定した内容と一致）
3. **ファイル名と中身を具体的に** → 「いい感じに作って」ではなく、何が欲しいか明確に

> 「曖昧なプロンプトは探索には良いが、実装には具体的な指示が必要」 -- 公式ベストプラクティス

Skillを作るときの重要なポイント：

### ちなみに: Skills・Hooks・Rules・Loopの違い

Claude Codeを自動化する仕組みは4つあります。混同しやすいので整理します。

| 仕組み | いつ動く | 何をする | 例え |
|-------|---------|---------|------|
| **Skills** | `/コマンド名` で呼ぶ、またはClaudeが自動検出 | よく使う指示セットを実行 | レシピ本 |
| **Hooks** | ツール実行の前後に自動発動 | 安全チェック・自動フォーマット | 火災報知器 |
| **Rules** | セッション開始時に自動読み込み | Claude Codeの判断基準を設定 | 社内ルールブック |
| **Loop** | 定期間隔で繰り返し実行 | 定期チェック・監視タスク | 定時パトロール |

**使い分けの考え方：**
- **毎回同じ操作をしている** → **Skills** にまとめる
- **操作の前後で必ずチェックしたい** → **Hooks** で自動化する
- **Claude Codeの振る舞いを全体的に変えたい** → **Rules** に書く
- **定期的に繰り返したい** → **Loop** で自動実行する

Step 3 で設定した deny リストは Hooks の一種です。`~/.claude/rules/` に置いた development.md や security.md は Rules です。

### ちなみに: カスタマイズ手段の全体像

```
何をしたい？
  ├─ よく使う指示を呼び出したい     → Skills
  ├─ 操作の前後で自動チェック       → Hooks
  ├─ Claude Codeの判断基準を設定    → Rules
  ├─ 定期的に繰り返し実行           → Loop
  ├─ 複雑なタスクを分担させたい     → サブエージェント
  ├─ 外部サービスと連携したい       → API or MCP
  └─ 出力のスタイルを変えたい       → 出力スタイル
```

### ちなみに: 外部サービスとの連携（API vs MCP）

| | API | MCP |
|--|-----|-----|
| 何が起きる | コードが作られる | Claude Codeが直接やり取りする |
| Claude Code閉じたら？ | **コードが残るから動き続ける** | **終わり** |
| 例えるなら | レシピを書いてもらう→次から自分で作れる | 料理を作ってもらう→次もまた頼む |

**MCPはClaude Codeへの「接続口」。Claude Codeが動いてないと何もできません。** APIはコードとして残るので、一度作ればClaude Codeなしでも動きます。
