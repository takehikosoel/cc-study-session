# 計画先行（Plan First）

## ルール

1. **コードを書く前に計画を提示する** — 「何を作るか」「どのファイルを変更するか」を先に見せる
2. **出力仕様がある場合は全パターンを先に確定する** — 通知、レポート、UIの出力内容を全て決めてから実装する
3. **ユーザーが「OK」と言うまでコードを書かない** — 計画の承認を待つ
4. **アーキテクチャの判断を推測しない** — デプロイ先、DB選定、フレームワーク選択はユーザーに確認する

## 計画に含めること

- 変更するファイル一覧
- 新規作成するファイル
- 依存関係（この順番で作る必要がある等）
- テスト方法
- 既存コードへの影響

## よくある間違い（過去の教訓）

- 要件定義だけ求められてるのにコードを書き始めた
- 通知の出力フォーマットを先に決めずに実装 → 後から全部書き直し
- デプロイ先をRailway全部と推測したが、実際はバックエンドだけだった
- ccmanagerの存在を知らずに実装を始めた
