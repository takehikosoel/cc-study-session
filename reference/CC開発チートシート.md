# Claude Code 開発チートシート

## 3段階フレームワーク（PSB）

### 1. Plan（計画）
- コードを書く前にAIと要件を深掘りする
- 「このプロジェクトを成功させるために答えるべき3つの質問は？」と聞く
- 成果物：製品要件（誰が/何を/どう解決）+ 技術要件（スタック明示）
- Plan Modeを使う（Shift+Tab×2）。「コードは書くな、読むだけ」と指示

### 2. Setup（環境整備）
- GitHubリポジトリ作成
- .env準備（APIキー等）
- CLAUDE.md作成（50行以内。プロジェクトの目標・ルール・ディレクトリ構造）
- 自動ドキュメント4ファイル：architecture.md / changelog.md / project_status.md / reference.md
- MCP接続（DB、GitHub等の外部ツール連携）

### 3. Build（実装）
- 「調査→計画→実装→テスト」の順。各フェーズ間で /clear
- Issue単位でClaudeに処理させる
- git worktreeで複数Claude並行開発

## コンテキスト管理（最重要）

- /clear を頻繁に使う。新タスクごとにリセット
- コンテキスト60%を超えない
- 自動compactionは避ける（不透明でエラーしやすい）
- 「Document & Clear」：計画と進捗を.mdに書き出し → /clear → 新セッションで.mdを読ませて続行
- 不要なMCPサーバーは /mcp で無効化

## CLAUDE.md の書き方

- 50行以内に収める（多すぎると全ての指示の遵守率が下がる）
- コードスタイルはlinterに任せる（CLAUDEに書かない）
- .claude/rules/ で分野別にルールを分離
- 長いドキュメントは@参照せず、パスだけ書いて「必要な時に読め」と指示

## 開発ワークフロー

### TDD（テスト駆動開発）
1. テストを先に書かせる
2. テストが失敗することを確認
3. 実装してテストを通す
4. 実装がテストに過剰適合していないかサブエージェントで検証

### カスタムコマンド
- .claude/commands/ にMarkdownでテンプレ保存
- $ARGUMENTS でパラメータ化
- シンプルに保つ（複雑にしすぎない）

### Hooks（自動化保証）
- PreToolUse：危険な操作をブロック（.envや.git/への書き込み禁止）
- PostToolUse：ファイル編集後に自動フォーマット/テスト実行
- プロンプトは「お願い」、Hooksは「保証」

### サブエージェント
- リサーチや検証を別エージェントに委任し、メインのコンテキストを節約
- 「4つの並行タスクで異なるディレクトリを調査して」
- ただし過度に専門化させない（メインが全体を見れなくなる）

## 実践Tips

- 音声入力を使う（タイピングの3倍速。PLAN.mdに思考をダンプ）
- thinkingレベルを使い分ける：基本→think / 設計→think hard / アーキ→ultrathink
- /rewind でチェックポイントに戻れる（コードのみ/会話のみ/両方）
- bashコマンド（rm, mv等）はチェックポイント対象外。注意
- セッションに /rename で名前をつけ、--resume で再開

## 並行開発パターン

- git worktreeで機能ブランチごとにClaude起動
- Writer + Reviewer パターン（1人が書き、もう1人がレビュー）
- 大規模リファクタは逐次（並行だとconflict解消で時間がかかる）
