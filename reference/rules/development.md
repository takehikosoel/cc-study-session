# CC開発ルール（全プロジェクト共通）

## 開発の進め方（PSBフレームワーク）

### 1. Plan（計画）— コードを書く前に
- Plan Modeで要件を深掘りする（Shift+Tab×2）
- 通知やUI等の出力は、全パターンの仕様を先に確定させてから実装に入る
- 「このプロジェクトを成功させるために答えるべき3つの質問は？」で始める

### 2. Setup（環境整備）
- CLAUDE.md を作成（50行以内。目的・構造・ルール）
- 機密情報は .env or Script Properties で管理（コードにハードコードしない）
- .claude/rules/ で分野別にルールを分離

### 3. Build（実装）
- 「調査 → 計画 → 実装 → テスト」の順。各フェーズ間で /clear
- 独立した作業はサブエージェントで並行実行
- 大規模リファクタは逐次（並行だとconflict解消で時間がかかる）

## コンテキスト管理（最重要）

- /clear を頻繁に使う。新タスクごとにリセット
- コンテキスト60%を超えない
- 「Document & Clear」：進捗をメモリに書き出し → /clear → 新セッションで続行
- 開発履歴は毎セッション終了時にメモリに記録する

## CLAUDE.md の書き方

- 50行以内に収める（多すぎると遵守率が下がる）
- 記載するのは3つだけ：Purpose（WHY）/ Repo map（WHAT）/ Rules（HOW）
- コードスタイルはlinterに任せる
- 長いドキュメントはパスだけ書いて「必要な時に読め」と指示

## 実装ルール

### コード品質
- 既存コードを読んでから修正する（読まずに提案しない）
- 過度な抽象化を避ける（3行の重複 > 早すぎる抽象化）
- 不要なファイルを作らない（既存ファイルの編集を優先）
- エラーハンドリングは外部境界（API、ユーザー入力）のみ

### 変数・定数の管理
- 大文字/小文字のキー名は統一する（プロジェクト内で混在させない）
- 角度やステータスのような定数はConfig等で一元管理する

### テスト
- 新機能はテスト関数を作って手動実行で確認
- 土日スキップ等の条件はテスト時にバイパスできるようにする

## サブエージェントの使い方

- 独立したファイルの実装は並行で投げる
- 各エージェントには完全な前提情報を渡す（特に定数の変更など）
- 依存関係がある場合は順次実行

## セッション管理

- /rename でセッションに名前をつける
- --resume で再開
- /rewind でチェックポイントに戻れる（bashコマンドは対象外）
- thinkingレベル：基本→think / 設計→think hard / アーキ→ultrathink
